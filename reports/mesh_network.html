Problem Statement

Design a routing algorithm for a mesh network modelled as an undirected weighted graph:

  • Each node has a unique integer ID in [1, 254] and a physical (x, y) position.
  • An edge exists between two nodes iff their distance is ≤ 50 metres.
  • Each node can measure the RSSI of received messages for link-quality estimation.
  • The network topology is unknown at initialisation — no routing tables pre-loaded.
  • The source must reach a destination that is not a direct neighbour.

② Comparative Analysis & Selection

  • Adjacency matrix: simple to implement but no native node-attribute storage; O(N²) space even for sparse networks — wasteful for N ≤ 254.
  • Adjacency list: lightweight and O(N+E) space, but all attribute management (position, RSSI, weight) is manual and error-prone.
  • NetworkX: richly attributed nodes and edges, built-in BFS/Dijkstra, excellent matplotlib integration — ideal prototype choice. Used in the project ZIP.
  • Boost Graph / igraph: high performance but require C++ expertise or native bindings; overkill for a proof-of-concept with N ≤ 254 nodes.
CHOSEN

Pure-Python adjacency list with BFS — zero external dependencies, runs entirely in Pyodide without additional packages. NetworkX is used in the project ZIP for production; the browser version reimplements the same interface with built-in dicts.

③ Free-Space Path-Loss Model & Solution Plan

The RSSI formula is derived from the Friis transmission equation [2]. Assuming unity antenna gains and a reference transmit power of 0 dBm:

ParameterValueJustification
Carrier frequency f2.4 GHzISM band (Zigbee / IEEE 802.15.4 [7])
Antenna gain Gt = Gr1 (0 dBi)Omnidirectional — conservative worst-case
Transmit power Pt0 dBm (normalised)Relative RSSI comparison only
Constant K147.5520·log₁₀(c / 4π) with d in metres, f in Hz
Edge weight1 (hop count)Routing objective is minimum hops → unit-weight BFS is optimal

Why weight = 1? The problem statement explicitly requests a route with a minimum number of hops. Assigning unit weight to every edge turns the shortest-path problem into a BFS problem, guaranteeing that the AODV RREQ flood finds the path crossing the fewest intermediate nodes — minimising latency, energy, and collision probability.

Solution plan: a Node class storing id, (x,y) position, and a neighbour table {id → RSSI dBm}; a MeshNetwork class wrapping an adjacency list with add_node, build_edges (iff distance ≤ 50 m), and bfs methods; and an ASCII canvas renderer for topology visualisation.

    2. 
    
  2. Receivers record sender ID, measured RSSI, and timestamp in their NeighbourTable.
    3. 
    
  3. Entries older than T_TIMEOUT = 15 s (3 missed HELLOs) are expired — link failure detected.
    4. 
    
  4. Monotonic seq allows duplicate detection in multi-path topologies.

Q3
Minimum-Hop Routing · AODV-inspired

  
① State of the Art

  

    
Algorithm / ProtocolTopology needed?Hop-optimal?Reactive?

    
      
Dijkstra (centralised)Full graphYesNo (proactive)

      
Bellman-Ford (dist. vector)1-hop onlyYes (converges)No

      
DSR [5]None at startYesYes — full path in header

      
AODV (RFC 3561) [1]None at startYes (BFS-like)Yes — on-demand

      
Geographic routingNode positionsNo (local minima)Yes

      
FloodingNoneYes (worst-case)Yes

    
  

  
② Comparative Analysis & Selection

  
    
    
  • Dijkstra / Bellman-Ford: require global or converged topology — impossible at initialisation when no routing tables are pre-loaded.
    • 
    
  • Flooding: always finds a path but generates O(N) redundant transmissions per packet — unacceptable overhead.
    • 
    
  • Geographic routing: requires node position knowledge not guaranteed in this problem; susceptible to local minima voids.
    • 
    
  • DSR [5]: stores the full path in every data packet header — overhead grows linearly with path length, costly for long chains.
    • 
    
  • AODV [1]: broadcasts RREQ only when needed; RREP unicasts the shortest path back. Hop-optimality is guaranteed because the first RREP to arrive at the source has crossed the fewest hops — the BFS property of simultaneous flooding with unit weights.
    • 
  

  
CHOSEN
AODV-inspired reactive protocol [1] — reactive (no topology pre-knowledge); BFS-optimal hop count; low steady-state overhead; standard for mesh sensor networks [4].

  
Note: This implementation uses source-routed paths (similar to DSR) rather than per-node routing tables as in RFC 3561 AODV. This simplification maintains AODV's route-discovery semantics (RREQ flood + RREP unicast) while using explicit path headers for forwarding.

  
③ Protocol Operation

  
Phase 1 — Route Discovery: source broadcasts RREQ(src, dst, seq, hops=0, path=[src]). Each node increments hops, appends its ID to path, and re-broadcasts (duplicate-suppressed by a seen_rreqs set). The destination sends a unicast RREP back along the recorded path, carrying the complete hop-optimal route.

  
Phase 2 — Data Forwarding: source installs the route from the RREP and forwards DATA packets hop-by-hop. Intermediate nodes also acquire reverse-path entries during RREQ propagation — these are free routing table entries obtained as a by-product of the flood.

③ Solution Planning

  
The custom DES builds on the AODV-inspired protocol from Q3, adding timed events for HELLO broadcasts, route discovery with TTL-scoped flooding, and data delivery with hop-by-hop forwarding. The SimNode class encapsulates per-node state (routing table, neighbour list, event log) and a unified run_scenario() driver function provides consistent execution, logging, and metric reporting across all test configurations.

Future extensions could incorporate: RSSI-weighted routing for link-quality-aware path selection (preferring stronger signals over raw hop count), energy-aware forwarding for battery-constrained nodes, and message authentication to prevent route-poisoning attacks in adversarial deployments.
